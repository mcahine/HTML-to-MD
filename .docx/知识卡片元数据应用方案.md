# 知识卡片元数据应用方案

> 本文档面向项目维护者与 AI 编码助手，描述项目中 Markdown 文件 YAML 元数据的结构、现状及可利用方向。

---

## 1. 背景与现状

### 1.1 元数据来源

本项目处理的 `.md` 文件存在两套并存的 YAML front matter：

- **基础型元数据**：由 `html_converter_base.py` 在 HTML→MD 转换时自动生成，包含 `title, author, date, source, url, converted_at` 等字段。
- **知识卡片型元数据**：依据《全自动知识卡片元数据生成规范 (v1.0)》由 AI 辅助生成，在基础型之上扩展了 `tags, category, description, aliases, related_concept, problem, summary, next_step, suggestion, audience, difficulty, rating, AI title` 等字段。

### 1.2 覆盖度现状

- 早期转换的文档通常**仅有基础型**元数据（约 6 个字段）。
- 近期转换且经过 AI 辅助补全的文档已具备**完整知识卡片型**元数据（约 17~19 个字段）。
- 两类文档在项目中共存，形成"半结构化"的知识库状态。

## 2. 元数据字段总览

| 字段名 | 类型 | 来源 | 用途定位 |
|--------|------|------|---------|
| `title` | 字符串 | 基础 | 文章原始标题 |
| `author` | 字符串 | 基础 | 作者名 |
| `date` | 日期 | 基础 | 文章发布日期 |
| `source` | 字符串 | 基础 | 来源网站 |
| `url` | URL | 基础 | 原始网页地址 |
| `converted_at` | 时间戳 | 基础 | 本项目转换时间 |
| `aliases` | 字符串 | 知识卡片 | 机器可读的短标识，用于文件名与内部链接 |
| `AI title` | 字符串 | 知识卡片 | AI 生成的、更适合人类阅读的重构标题 |
| `description` | 字符串 | 知识卡片 | 概念性摘要，说明主题与范围（≤50 字） |
| `summary` | 字符串 | 知识卡片 | 精华要点，3~5 条核心收获，用分号分隔 |
| `problem` | 多行文本 | 知识卡片 | 1~3 个文档试图解决的核心问题 |
| `related_concept` | 字符串 | 知识卡片 | 外部关联概念，用 `/` 分隔（最多 5 个） |
| `category` | 字符串 | 知识卡片 | 归档分类，如"专业技术/产品与设计" |
| `tags` | 列表 | 知识卡片 | 主题标签体系，如"编程与架构/微服务" |
| `audience` | 字符串 | 知识卡片 | 目标读者画像 |
| `difficulty` | 枚举 | 知识卡片 | 阅读难度：入门/进阶/专业/专家/行业 |
| `rating` | 整数 | 知识卡片 | 价值评分 1~5 |
| `next_step` | 字符串 | 知识卡片 | 前瞻性后续行动建议 |
| `suggestion` | 字符串 | 知识卡片 | 笔记完善建议 |

## 3. 应用场景详解

### 3.1 Obsidian 内原生能力（Dataview + 图谱）

#### 3.1.1 动态精华阅读表

利用 `rating`、`category`、`difficulty` 做过滤排序，在 Obsidian 中生成"高价值文章速览"。

示例 Dataview 查询：

```dataview
table AI title, category, difficulty, rating, summary
from ""
where rating and category
sort rating desc, date desc
limit 20
```

#### 3.1.2 待完善笔记清单

利用 `suggestion` 字段聚合所有"需要补充"的笔记，形成知识库治理的入口。

```dataview
table AI title, category, suggestion
from ""
where suggestion and suggestion != "内容已较完整，无需补充"
sort date desc
```

#### 3.1.3 图谱局部聚焦

传统 Obsidian 图谱在 1000+ 篇笔记时会过于稠密。利用 `tags` 和 `related_concept` 做过滤，可实现：

- 只显示带有 `人工智能` tag 的文章子图；
- 只显示与 `扩散模型` 相关的概念网络。

#### 3.1.4 别名快速跳转

`aliases` 字段提供了除标题外的第二检索入口。在 Obsidian 中输入 `Qwen3.6` 或 `AbortController`，即使标题未包含这些词，也能通过别名精准定位文章。

### 3.2 自动化语义导航生成

现有导航脚本（`.code/配置与导航/导航.py`）按**文件系统目录（网站域名）**组织内容，这是"来源维度"。元数据可构建**"知识维度"**的导航：

#### 3.2.1 按 category 聚类导航

打破网站边界，将"专业技术"类文章跨目录聚合：

```
专业技术
├── 编程与架构
│   ├── AbortController 底层原理（来自掘金）
│   └── Qwen3.6-Plus 刷新纪录（来自 IT之家）
├── 人工智能
│   └── LLaDA-MoE 扩散语言模型（来自机器之心）
```

#### 3.2.2 按 difficulty 分级阅读路径

为不同水平读者生成定制化入口：

- **入门读者**：只展示 `difficulty: 入门` 的文章，屏蔽术语密集的内容；
- **专业读者**：展示 `difficulty: 专业` 以上且 `rating >= 4` 的精华。

#### 3.2.3 按 problem 问答式导航

将 `problem` 字段抽离为"问题索引页"：

```
常见问题索引
├── 冷启动的根源是什么？
│   └── → 指向《Serverless 冷启动优化》
├── 扩散语言模型如何反向去噪？
│   └── → 指向《LLaDA-MoE 扩散语言模型》
```

**实施建议**：可编写独立脚本 `generate_semantic_nav.py`，读取所有 `.md` 的 YAML，生成一个静态的 `语义导航.md`，无需启动 HTTP 服务即可在 Obsidian 中浏览。

### 3.3 AI 检索增强（RAG）

在将知识库接入大模型（本地或 API）时，直接拿全文做 RAG 会面临噪声大、召回不准的问题。知识卡片元数据恰好可充当**精准检索层**：

| 元数据字段 | RAG 角色 | 说明 |
|-----------|---------|------|
| `summary` | **摘要向量** | 比全文更浓缩、语义更干净，适合生成 embedding 做语义检索 |
| `description` | **主题过滤器** | 粗排阶段快速判断文章是否属于目标主题 |
| `problem` | **问题对齐器** | 用户提问与 `problem` 字段做相似度匹配，精准定位"回答此问题的文章" |
| `related_concept` | **关联扩展器** | 检索到一篇文章后，自动沿概念网络推荐同簇其他文章 |
| `tags` | **分类路由器** | 大模型根据 tag 决定调用哪个领域的知识子库，降低跨域干扰 |

**实现思路**：构建双层检索架构——
1. **元数据索引层**：只将 `summary`、`description`、`problem`、`tags` 向量化存入向量数据库；
2. **全文层**：在元数据层命中后，再取对应文章的全文做细粒度阅读。

这样可在不牺牲准确度的前提下，大幅降低 embedding 成本和检索噪声。

### 3.4 长期知识运营与治理

#### 3.4.1 阅读回顾周报/月报

利用 `date` + `summary` + `category`，可自动生成：

> "本周你阅读了 12 篇文章，其中 5 篇属于'人工智能'，核心价值包括：扩散语言模型引入 MoE 架构可将推理速度提升 3 倍……"

#### 3.4.2 知识缺口诊断

批量扫描 `suggestion` 字段，可发现：

- 某类主题（如"项目管理"）的笔记普遍被建议"补充案例"，说明该领域你的知识尚未落地；
- 某个 `category` 下的文章 `rating` 普遍偏低，说明你可能在阅读低质量来源。

#### 3.4.3 重复摄入检测

通过比对 `aliases` 和 `related_concept`，可发现：

- 两个不同网站的文章若 aliases 高度重合，很可能是同一事件的不同报道；
- 可据此建议"保留一篇高 rating 的，其余归档"。

#### 3.4.4 核心知识沉淀

定期将 `rating: 5` 的文章抽取出来，形成《个人核心知识手册》。这些文章具有"深刻洞察、罕见经验或高度浓缩的思维框架"，值得反复引用。

## 4. 实施路线图

### 阶段一：元数据补全（当前最高优先级）

- [ ] 编写批量元数据补全脚本，识别仅有基础型元数据的文档；
- [ ] 接入 AI API（或本地模型），按规范为存量文档生成知识卡片型元数据；
- [ ] 对新生成的元数据做抽样人工校验，确保 `aliases`、`tags`、`category` 符合规范。

### 阶段二：Obsidian 工具化

- [ ] 在 `.obsidian/` 中预设 Dataview 查询模板（精华表、待完善清单、按难度分级表）；
- [ ] 编写 `generate_semantic_nav.py`，按 `category` / `tags` / `problem` 生成静态语义导航页；
- [ ] 将语义导航页加入 Obsidian 的默认星标或日记模板。

### 阶段三：AI 检索接入

- [ ] 将 `summary` + `description` + `problem` 抽取为纯文本索引；
- [ ] 接入本地向量数据库（如 Chroma、Milvus Lite），构建元数据级语义检索；
- [ ] 开发 CLI 工具 `knowledge_query.py`，支持命令行问答。

### 阶段四：知识运营自动化

- [ ] 编写 `weekly_review.py`，按周聚合阅读内容并生成回顾 Markdown；
- [ ] 编写 `knowledge_gap.py`，分析 `suggestion` 与 `rating` 分布，输出治理报告；
- [ ] 探索与外部工具（Notion、Heptabase、Logseq）的 YAML 同步方案。

## 5. 注意事项

- **字段一致性**：`tags` 与 `category` 的值必须严格来自规范文档中的预设枚举，禁止随意创造新值，否则聚合与导航会失效。
- **YAML 格式安全**：`problem`、`summary` 等字段含中文标点与特殊字符时，注意 YAML 转义（如多行文本使用 `|-` 格式）。`polish_markdown.py` 已包含 YAML 规范化修复，应定期运行。
- **性能考量**：当文档数量超过 5000 篇时，全量扫描 YAML 的速度会下降。建议后续引入增量索引机制（基于 `converted_at` 或 `date modified`）。
